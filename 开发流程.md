# Telegram Mini App 视频售卖系统开发流程

## 1. 项目目标

开发一个通过 Telegram Mini App 售卖私密视频访问权限的系统。

技术组合：

- Telegram Mini App：用户购买和观看入口。
- Node.js + TypeScript：后端接口、鉴权、订单、播放授权。
- Cloudflare Stream：视频托管和播放。
- Cloudflare Stream signed token：防止公开视频链接被复制传播。
- 播放器层水印：只在播放器上覆盖水印，不使用 FFmpeg 重新生成视频。

MVP 的目标不是绝对防录屏，而是：

- 防止普通用户直接保存或转发链接。
- 让泄露视频可以通过订单号追踪。
- 提高二次售卖成本。
- 保留播放记录，方便风控和封禁。

## 2. 安全边界说明

播放器水印适合作为第一版方案，因为 Telegram 手机端 Mini App 对普通用户来说不容易像网页一样直接打开开发者工具。

但前端水印不是强安全边界，仍然无法彻底防止：

- 用另一台设备拍屏。
- 系统级录屏。
- 高级用户抓包分析播放请求。
- 在可控浏览器环境里隐藏 DOM/CSS 水印。
- 对录屏画面进行裁剪、遮挡或重编码。

所以系统的核心策略是：

- 短期播放授权。
- 不开放下载。
- 每个订单独立水印。
- 播放行为记录。
- 发现泄露后能定位订单并封禁。

## 3. MVP 总体架构

```text
Telegram Bot
    |
    | 打开 WebApp 按钮
    v
Telegram Mini App 前端
    |
    | 携带 Telegram initData 请求后端
    v
Node.js API 后端
    |
    | 校验 Telegram 身份
    | 检查订单和观看权限
    | 创建播放 session
    | 生成 Cloudflare Stream signed token
    v
Cloudflare Stream Player
    |
    | 播放视频
    | 覆盖订单号水印 + 官方水印
    v
用户观看视频
```

## 4. 水印方案

水印只做在播放器层，不烧录进视频文件。

### 4.1 用户水印

用户水印只显示订单号。

示例：

```text
A8K29Q7M
```

这样做的原因：

- 不直接暴露用户 Telegram ID 或用户名，隐私感更好。
- 画面文字更短，移动端显示更干净。
- 泄露后仍然可以通过后台订单记录追踪到购买者。

后端必须保存以下映射关系：

```text
订单号 -> Telegram 用户 -> 视频 -> 购买时间 -> 播放记录
```

### 4.2 官方水印

系统还需要一个官方水印，用来标识内容来源。

示例：

```text
Official
你的品牌名
```

上线前需要把官方水印替换成你的品牌名、频道名或平台名。

### 4.3 水印位置规则

建议规则：

- 官方水印固定在右上角或左上角。
- 订单号水印每 8-15 秒移动一次。
- 订单号水印不要只放角落，要偶尔出现在画面中间区域。
- 水印透明度建议在 `0.18-0.35` 之间。
- 水印层设置 `pointer-events: none`，避免挡住播放器控制按钮。
- 水印必须在视频画面区域内，不能只放在播放器外层。

### 4.4 MVP 最低要求

第一版至少实现：

- 官方水印固定显示。
- 订单号水印动态移动。
- 水印在手机屏幕上不遮挡主要控制按钮。
- 播放 session 和订单号在后台可追踪。

## 5. 用户主流程

1. 用户打开 Telegram Bot。
2. Bot 提供进入 Mini App 的按钮。
3. 用户打开 Mini App。
4. 前端读取 Telegram `initData`。
5. 前端调用 `/api/auth/telegram`。
6. 后端校验 `initData` 签名和 `auth_date`。
7. 用户进入视频列表。
8. 用户购买视频，或打开已购买视频。
9. 后端检查用户是否拥有观看权限。
10. 后端创建播放 session。
11. 后端生成 Cloudflare Stream signed token。
12. 前端加载 Cloudflare Stream 播放器。
13. 前端覆盖官方水印和订单号水印。
14. 前端定期上报播放心跳。

### 6.4 API 接口

```text
POST /api/auth/telegram
- 入参：initData
- 出参：用户信息、后端 session token

GET /api/videos
- 出参：视频列表

GET /api/videos/{video_id}
- 出参：视频详情、购买状态、观看状态

POST /api/videos/{video_id}/play
- 要求：已登录 Telegram 用户
- 检查：用户是否有观看权限
- 创建：play_session
- 出参：Cloudflare 播放地址/token、订单号、官方水印、session_code

POST /api/play-sessions/{session_code}/events
- 记录：play、pause、seek、heartbeat、ended

POST /api/orders
- 创建待支付订单

POST /api/payments/webhook
- 校验支付平台回调
- 标记订单已支付
- 发放观看权限
```

### 6.5 Telegram 身份校验

后端必须：

- 解析 Telegram Mini App `initData`。
- 使用 Telegram Bot Token 校验 hash。
- 拒绝过期的 `auth_date`。
- 根据校验通过的数据创建或更新用户。
- 不信任前端直接传来的用户 ID、用户名或订单信息。

建议过期时间：

```text
普通登录：86400 秒
敏感操作：300 秒
```

## 7. Cloudflare Stream 开发计划

### 7.1 视频设置

每个受保护视频都应该：

- 开启 `requireSignedURLs`。
- 不开放公开视频播放。
- 不开放 MP4 下载。
- 只能通过后端生成的短期 signed token 播放。

### 7.2 Token 策略

MVP 建议：

```text
有效期：5-15 分钟
downloadable：false
每次播放重新生成
```

如果 token 过期但用户仍然拥有权限，前端可以重新请求播放 token。

### 7.3 视频上传方式

MVP 阶段可以先手动上传：

1. 在 Cloudflare Dashboard 上传视频。
2. 复制 Cloudflare Stream video UID。
3. 写入 `videos` 表或 seed 脚本。
4. 后台暂时不做复杂上传管理。

后续再开发 admin 上传和视频管理。

## 8. 前端开发计划

### 8.1 技术栈

- `@tma.js/create-mini-app` 脚手架
- React
- TypeScript
- Vite
- `@tma.js/sdk-react`
- Cloudflare Stream iframe/player embed

### 8.2 Mini App 框架选择

前端推荐直接使用 Telegram Mini Apps 生态里的 `@tma.js/create-mini-app`。

推荐选择：

```text
Language: TypeScript
SDK: tma.js
Framework: React.js
Bundler: Vite
```

理由：

- 能直接生成 Telegram Mini App 项目结构。
- 支持 React、Next.js、Vue、Solid 等模板，MVP 选 React 更轻。
- Vite 启动快，适合移动端页面开发和调试。
- `@tma.js/sdk-react` 提供 React hooks，可以读取启动参数、主题、Back Button、Viewport 等 Telegram 能力。
- 后端可以配套使用 `@tma.js/init-data-node` 校验 Telegram 身份，全栈都在 Node.js 生态内。

### 8.3 页面

MVP 页面：

- 视频列表页。
- 视频详情页。
- 购买状态页。
- 播放器页。
- 无权限/已过期提示页。

### 8.4 播放器页行为

播放器页需要：

- 先向后端请求播放授权。
- 只有拥有权限时才加载 Cloudflare Stream 播放器。
- 显示官方水印。
- 显示移动的订单号水印。
- 每 15-30 秒上报播放心跳。
- token 过期时重新请求。
- 手机端布局不遮挡播放器控制区域。

### 8.5 推荐 DOM 结构

```text
player-shell
- stream-player
- official-watermark
- order-watermark
```

关键 CSS：

```text
position: absolute
pointer-events: none
user-select: none
z-index: 高于播放器
```

订单号水印的位置由前端状态控制，定时更新。

## 9. 支付方案

支付可以分阶段做。

### 9.1 MVP 阶段

先做模拟支付：

- 用户点击购买。
- 后端创建订单。
- 开发环境提供手动标记支付成功的接口或脚本。
- 支付成功后发放观看权限。

这样可以先把播放链路打通。

### 9.2 正式支付

可选方案：

- Telegram Bot Payments。
- Stripe。
- 加密货币支付网关。
- 其他第三方支付平台。

正式支付必须满足：

- 后端校验支付回调签名。
- 只通过可信支付回调发放权限。
- 不允许前端自己告诉后端“我已付款”。

## 10. 管理后台计划

MVP 可以先不做完整后台。

最低管理能力：

- seed 脚本创建视频。
- 数据库查看订单。
- 手动撤销观看权限。
- 手动标记泄露订单。

后续管理后台：

- 创建和编辑视频。
- 绑定 Cloudflare Stream 视频。
- 查看用户。
- 查看订单。
- 查看播放记录。
- 撤销权限。
- 查看可疑播放行为。

## 11. 风控和滥用检测

第一版记录以下信号：

- 同一个订单短时间内创建太多播放 session。
- 同一个订单出现多个 IP。
- 同一个订单出现多个设备或 user-agent。
- 同一个订单同时播放。
- 同一个订单长时间循环播放。

后续处理方式：

- 提醒用户。
- 临时冷却。
- 暂停播放。
- 撤销权限。
- 人工审核。

## 12. 开发里程碑

### 阶段 1：项目初始化

- 创建 `backend/` Node.js + TypeScript + Fastify 项目。
- 使用 `@tma.js/create-mini-app` 创建 `frontend/` Mini App 项目。
- 创建 `.env.example`。
- 创建本地启动脚本。
- 保留中文开发流程文档。

### 阶段 2：Telegram 登录

- 实现 `initData` 校验。
- 创建 users 表。
- 前端接入 Telegram WebApp 初始化。
- 页面显示当前登录用户状态。

### 阶段 3：视频列表

- 创建 videos 表。
- 添加 seed 示例视频。
- 前端显示视频列表和详情。

### 阶段 4：订单和权限

- 创建 orders 和 entitlements 表。
- 实现模拟购买。

### 阶段 5：Cloudflare Stream 播放

- 后端生成 signed token。
- 前端加载 Cloudflare Stream 播放器。

- 添加官方水印。
- 添加订单号动态水印。

- 创建 play_sessions。

### 阶段 8：正式支付

- 选择支付平台。
- 创建订单接口。
- 接入支付回调。
- 支付成功后发放观看权限。

### 阶段 9：部署上线

- 部署后端。
- 部署前端。
- 配置 Telegram Bot WebApp URL。
- 配置 Cloudflare Stream。
- 配置生产数据库。
- 做完整购买和播放测试。

## 13. 本地开发需要你提供的配置

下面是我后续开发和联调时需要你准备或提供的信息。

### 13.1 Telegram 配置

必需：

```text
TELEGRAM_BOT_TOKEN=
```

获取方式：

1. 在 Telegram 找到 `@BotFather`。
2. 创建 Bot 或使用已有 Bot。
3. 复制 Bot Token。

还需要你后续设置：

- Bot 的 Mini App / WebApp 入口按钮。
- Mini App 的线上 URL，部署后才能正式配置。

本地开发时，如果要在 Telegram 里真实打开 Mini App，通常还需要一个 HTTPS 开发地址，例如 ngrok、Cloudflare Tunnel 或正式测试域名。

### 13.2 Cloudflare Stream 配置

必需：

```text
CLOUDFLARE_ACCOUNT_ID=
CLOUDFLARE_API_TOKEN=
CLOUDFLARE_STREAM_SIGNING_KEY_ID=
CLOUDFLARE_STREAM_SIGNING_PRIVATE_KEY=
```

还需要：

```text
测试视频的 Cloudflare Stream video UID
```

Cloudflare 侧需要你确认：

- 已开通 Cloudflare Stream。
- 视频已上传。
- 视频开启 `requireSignedURLs`。
- API Token 有 Stream 相关权限。
- signed URL key 已创建。

### 13.3 官方水印配置

必需：

```text
OFFICIAL_WATERMARK_TEXT=
```

例如：

```text
OFFICIAL_WATERMARK_TEXT=你的品牌名
```

用户水印不需要手动配置，系统会从订单号生成。

### 13.4 数据库配置

数据库按你习惯的 MySQL 8.0 开发。

```text
DATABASE_URL=mysql://user:password@127.0.0.1:3306/tgwebapp
```

你需要提供或确认：

```text
MYSQL_HOST=127.0.0.1
MYSQL_PORT=3306
MYSQL_DATABASE=tgwebapp
MYSQL_USER=
MYSQL_PASSWORD=
```

如果本地还没有库，需要先创建：

```sql
CREATE DATABASE tgwebapp CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

### 13.5 前端配置

本地需要：

```text
VITE_API_BASE_URL=http://localhost:8000
VITE_TELEGRAM_BOT_USERNAME=
```

如果使用 HTTPS 隧道测试：

```text
VITE_API_BASE_URL=https://你的后端测试域名
```

### 13.6 纯 Node.js 开发和部署说明

本项目不再使用 LAMP，不依赖 PHP 或 Apache。

这个项目的推荐方式是：

- 前端开发阶段使用 Node.js 运行 Vite 开发服务器。
- 后端开发阶段使用 Node.js 运行 Fastify API 服务。
- 前端上线时执行 `npm run build`，生成静态文件。
- 生产环境可以用 Node.js 后端同时提供 API，并托管前端静态文件。
- 也可以使用 Nginx 做反向代理和 HTTPS 入口。
- MySQL 8.0 作为数据库。
- 进程管理推荐使用 PM2 或 systemd。

## 15. MVP 验收标准

第一版完成时，需要满足：

- 用户可以从 Telegram 打开 Mini App。
- 后端能校验 Telegram `initData`。
- 用户可以看到视频列表。
- 用户只能观看已购买视频。
- 后端可以生成短期 Cloudflare Stream 播放 token。
- 视频能在 Mini App 播放。
- 播放器上显示官方水印。
- 播放器上显示移动的订单号水印。
- 播放事件可以记录到后端。
- 未购买或权限过期用户不能获取播放 token。
